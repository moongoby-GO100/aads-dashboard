Layer 1: Frontend Model Selection

  • auto/mixture: Intent Classification → Auto Model Decision (Layer 2~3)
  • Specific Model: model_override → Governance Bypass → Layer 4 Direct
  • Sort: mixture top → pinned → favorite → display_order

Layer 2: Intent Classification

  • Primary: qwen-turbo (LiteLLM), temperature 0.1, 5s timeout
  • Redis Cache: TTL 60s, SHA256(workspace + msg[:100] + context)[:16]
  • Correction 4-Step: PC followup → casual/greeting command fix → tool history → discussion guard
  • Keyword Fallback: 42 rules on LLM failure (HEAD → SEARCH → TAIL priority)

Layer 3: Model Decision (Dynamic Cascading)

1. model_override normalize ("mixture"/"auto"/empty → None)
2. model = override || intent_result.model
3. empty model → _get_default_llm_model_from_db() → Current: codex:gpt-5.5
4. provider:model split → alias resolve → registered model alias mapping
5. Dynamic Cascading (model_locked=False):
   → _resolve_governed_intent_model()
   → greeting/casual → claude-haiku downgrade
   → search/browser → claude-sonnet downgrade
6. Per-intent temperature (0.1~0.25)

11. 현황 진단: 문제점 및 리스크

🔴 Critical

C-1. Governance DB Primary Disabled
intent_policies_db_primary=False — 7 DB policies in Shadow Mode only. Actual routing uses hardcoded legacy rules. DB policy changes not reflected in real-time, limiting operational flexibility.
C-2. Insufficient Intent Downgrade Coverage
Only 9 intents downgraded by legacy rules. Remaining 61 intents use default high-cost models. Simple query intents like task_history, cost_report, file_read still use claude-opus.

🟠 Warning

W-1. Model Set Hardcoding
_GEMINI_MODELS, _GROQ_MODELS, _CODEX_MODELS etc. hardcoded in model_selector.py. New model add/remove requires code deployment. Manual sync with DB llm_models table only.
W-2. Intent Classifier Single Point of Failure
Intent classification depends on qwen-turbo (LiteLLM) single model. On LiteLLM failure, only keyword fallback available — 42 rules cover only 42 of 70 intents (28 uncovered).
W-3. Legacy/DB Rule Duplication
_resolve_governed_intent_model() runs both legacy and DB in parallel for comparison. Unintended mismatches accumulate in governance_audit_log, increasing management complexity.
W-4. Rigid Same-Grade Fallback
_SAMEGRADE_FALLBACK dict hardcoded. New competitive models (kimi-k2.6, minimax-m2.7) not auto-reflected.

🟡 Notice

N-1. No Proactive Usage-Based Routing
No auto-switch to alternative models when Claude Max 5h/1w usage exceeds 80%. Currently relies on 429 error then fallback chain, causing user experience delay.
N-2. No Governance Audit Dashboard
governance_audit_log data accumulates but no visualization UI. Difficult to identify legacy-DB mismatch patterns.
N-3. No Intent Classification Accuracy Metrics
Redis caching and 4-step correction exist, but no quantitative accuracy measurement. Cannot track misclassification rate.
